\hypertarget{expanding-the-motoko-manifesto-architectures-for-self-modifying-ai-agents}{%
\section{Expanding the Motoko Manifesto: Architectures for
Self-Modifying AI
Agents}\label{expanding-the-motoko-manifesto-architectures-for-self-modifying-ai-agents}}

\emph{Disclaimer: The following report discusses experimental software
architectures, automated code generation, and AI-driven system
modifications. This content is provided strictly for informational and
educational purposes only. It does not constitute professional
engineering, security, or architectural advice. Implementing
self-modifying autonomous agents in production environments carries
significant security, stability, and operational risks.}

\hypertarget{executive-summary}{%
\subsubsection{Executive Summary}\label{executive-summary}}

The transition from human-authored syntax to AI-driven code generation
requires a fundamental reimagining of software architecture. This report
directly addresses the six core components of this transition:

\begin{itemize}
\tightlist
\item
  \textbf{Evolvable and Phoenix Architectures:} Code is transitioning
  into an ephemeral build artifact. The true durable assets of software
  organizations are now the specifications, tests, and ``design
  provenance'' from which AI agents can continuously regenerate flawless
  systems from scratch.
\item
  \textbf{Formal Verification:} To ensure self-modifying code does not
  break its own underlying logic, systems like Motoko are relying on
  advanced mathematical proofs (e.g., Z3 contracts and Separation Logic)
  to constrain the probabilistic outputs of Large Language Models (LLMs)
  into verifiable bounds.
\item
  \textbf{Neurosymbolic Reasoning:} Opaque neural networks are
  insufficient for autonomous operations requiring strict compliance.
  Hybrid architectures, such as Tsetlin Machines, are being integrated
  to pair neural creativity with the interpretable, deterministic logic
  necessary for human stewardship.
\item
  \textbf{Deterministic Simulation and Evaluation:} Evaluating
  self-modifying agents requires moving beyond static code-completion
  tasks. Modern testing utilizes deterministic simulation and complex,
  multi-turn operational benchmarks (like Terminal-Bench 2.0) to
  evaluate trajectory quality, tool execution, and error-recovery over
  long horizons.
\item
  \textbf{Human-AI Stewardship:} The role of the human engineer is
  shifting from author to steward. This entails utilizing frameworks
  where agents are context-aware, capable of ``soft refusals'' when
  encountering ambiguity, and actively monitored for behavioral shifts
  induced by the ``Observer Effect.''
\item
  \textbf{AI-Native Languages and Extensions:} Traditional programming
  languages optimize for human ergonomics. Emerging agent ecosystems
  rely on AI-native languages (like AILANG) optimized for deterministic
  reasoning, paired with heavily extensible, secure plugin architectures
  spanning platforms like Aider, Cursor, Devon, and SWE-agent.
\end{itemize}

\begin{center}\rule{0.5\linewidth}{0.5pt}\end{center}

Research suggests that the software engineering paradigm is entering a
profound transition phase, shifting from manual human implementation to
AI-driven generation and stewardship. It seems likely that
specifications, intent, and rigorous tests will gradually replace
implementation code as the primary durable assets of software
organizations. The evidence leans toward hybrid neurosymbolic
approaches---pairing the probabilistic reasoning of Large Language
Models (LLMs) with the deterministic boundaries of formal
verification---as the most viable path to making AI-authored,
self-modifying code reliable in production environments.

The Motoko Manifesto outlines a bold vision for this future through the
lens of ``Motoko,'' an experimental AI coding agent harness. In this
proposed architecture, AI systems autonomously plan, execute, test, and
refactor code in continuous loops, constrained by mathematically
verifiable logic and explicit architectural intent. Human operators are
thus elevated from authors of syntax to stewards of system boundaries.

To achieve this level of autonomous operation, the manifesto advocates
for evolutionary software designs, verifiable AI-native languages,
deterministic evaluation, and explicit human-AI stewardship frameworks.
This comprehensive report expands on the ten interconnected themes of
the Motoko Manifesto, synthesizing current academic research, industrial
practices, and open-source developments to contextualize and critically
analyze the architecture of self-modifying AI agent systems.

\hypertarget{evolvable-and-phoenix-architectures-for-ai-authored-code}{%
\subsection{1. Evolvable and Phoenix Architectures for AI-Authored
Code}\label{evolvable-and-phoenix-architectures-for-ai-authored-code}}

The shift toward AI-authored code fundamentally alters how software
architectures are maintained. When human developers write code, they
optimize for readability, maintainability, and cognitive load. However,
when AI agents manage codebases, the architectural priorities shift
toward regenerability, modularity, and strict adherence to defined
fitness functions.

\hypertarget{the-dynamics-of-evolutionary-architecture}{%
\subsubsection{The Dynamics of Evolutionary
Architecture}\label{the-dynamics-of-evolutionary-architecture}}

The foundational premise of evolutionary software architecture, as
articulated by Neal Ford, Rebecca Parsons, and Patrick Kua, is that
systems must support guided, incremental change across multiple
dimensions
\textsuperscript{\protect\hyperlink{cite-1}{1},\protect\hyperlink{cite-2}{2}}.
This is achieved through the use of \textbf{fitness
functions}---automated, quantifiable tests that drive the evolution of a
software system in a desirable direction while protecting critical
architectural characteristics (such as security, latency, or decoupling)
from degrading over time
\textsuperscript{\protect\hyperlink{cite-2}{2},\protect\hyperlink{cite-3}{3}}.

When applied to AI-maintained codebases, evolutionary architecture acts
as a vital guardrail. An AI agent lacking strict boundaries may
inadvertently introduce tight coupling, creating a ``Big Ball of Mud''
anti-pattern where changes ripple destructively across the system
\textsuperscript{\protect\hyperlink{cite-3}{3},\protect\hyperlink{cite-4}{4}}.
By codifying constraints into fitness functions that run in deployment
pipelines, architects transition from being gatekeepers to guides,
ensuring that AI-driven self-modification naturally converges toward an
optimal structural state
\textsuperscript{\protect\hyperlink{cite-2}{2},\protect\hyperlink{cite-3}{3}}.

\hypertarget{the-phoenix-architecture-code-is-ephemeral}{%
\subsubsection{The Phoenix Architecture: Code is
Ephemeral}\label{the-phoenix-architecture-code-is-ephemeral}}

Building directly on evolutionary principles, Chad Fowler's ``Phoenix
Architecture'' posits a radical shift: code should be treated as a
disposable build artifact, while the specification (or intent) is the
true durable asset
\textsuperscript{\protect\hyperlink{cite-5}{5},\protect\hyperlink{cite-6}{6}}.

The Phoenix methodology is best understood through its core tenets:

\begin{itemize}
\tightlist
\item
  \textbf{The Deletion Test}: If a development team executes
  \texttt{rm\ -rf\ src/} (deleting the entire implementation) and cannot
  regenerate it flawlessly from tests and specifications, the system's
  meaning is stored in the wrong place
  \textsuperscript{\protect\hyperlink{cite-7}{7}}.
\item
  \textbf{Intent as Source of Truth}: Similar to how immutable
  infrastructure dictates that servers should be replaced rather than
  modified, immutable code dictates that implementation should be
  continuously regenerated from high-level specifications
  \textsuperscript{\protect\hyperlink{cite-5}{5},\protect\hyperlink{cite-6}{6}}.
\item
  \textbf{Evaluations as the Codebase}: The true intellectual property
  of a software system transitions from its syntax to the evaluations,
  constraints, and test suites that prove its correctness
  \textsuperscript{\protect\hyperlink{cite-5}{5}}.
\end{itemize}

This philosophy aligns with historical concepts in literate programming
and specification-driven development but relies on modern LLMs as the
``compiler'' that translates human intent into executable syntax
\textsuperscript{\protect\hyperlink{cite-5}{5},\protect\hyperlink{cite-7}{7}}.
While pure deletion tests remain largely anecdotal in enterprise
environments, emerging frameworks demonstrate that strict separation of
intent from execution dramatically reduces technical debt and
maintenance costs
\textsuperscript{\protect\hyperlink{cite-5}{5},\protect\hyperlink{cite-8}{8}}.

\hypertarget{design-provenance-vs.-craft-provenance}{%
\subsubsection{Design Provenance vs.~Craft
Provenance}\label{design-provenance-vs.-craft-provenance}}

To successfully regenerate code, a system must preserve not only what
was built, but \emph{why}. This introduces the distinction between
design provenance and craft provenance.

\textbf{Design provenance} refers to the overarching architectural
intent, the historical record of decisions, and the specifications that
dictate a system's constraints
\textsuperscript{\protect\hyperlink{cite-9}{9},\protect\hyperlink{cite-10}{10}}.
\textbf{Craft provenance}, conversely, involves the tacit knowledge,
manual optimizations, and specific implementation quirks historically
provided by human engineers
\textsuperscript{\protect\hyperlink{cite-9}{9},\protect\hyperlink{cite-11}{11}}.
In AI-authored systems, relying on craft provenance is dangerous, as
statistical models lack persistent tacit knowledge between sessions.
Consequently, systems must capture rigorous design provenance. Emerging
research proposes tracking this provenance through immutable meta-data
graphs or even blockchain-backed ledgers integrated directly into
Integrated Development Environments (IDEs)
\textsuperscript{\protect\hyperlink{cite-10}{10},\protect\hyperlink{cite-12}{12},\protect\hyperlink{cite-13}{13}}.
By maintaining an unbroken chain of design intent, AI agents can safely
delete and rewrite modules without losing the underlying architectural
logic.

\hypertarget{formal-verification-and-self-modifying-systems}{%
\subsection{2. Formal Verification and Self-Modifying
Systems}\label{formal-verification-and-self-modifying-systems}}

If an AI agent is granted the autonomy to modify its own underlying
code, traditional testing methodologies are insufficient to guarantee
safety. The Motoko Manifesto leverages formal
verification---mathematical proofs of software correctness---to
constrain self-modification and ensure that the agent never breaks its
own foundational logic.

\hypertarget{verifying-self-modifying-code-smc}{%
\subsubsection{Verifying Self-Modifying Code
(SMC)}\label{verifying-self-modifying-code-smc}}

Historically, formal verification techniques such as Hoare logic (a
formal system using rigorous pre- and post-conditions to reason about
program correctness) and dependent type systems (type systems where a
type depends on a value, allowing properties of programs to be encoded
directly into their types) assumed that program code in memory was fixed
and immutable \textsuperscript{\protect\hyperlink{cite-14}{14}}. This
assumption breaks down in the context of Self-Modifying Code (SMC),
which mutates its own instructions at runtime.

To overcome this, academic frameworks like \textbf{GCAP (General Code
Assembly Program)} extend Separation Logic (an extension of Hoare logic
specifically designed to simplify reasoning about programs that mutate
shared data structures or memory pointers) to treat machine instructions
uniformly as mutable data structures
\textsuperscript{\protect\hyperlink{cite-14}{14}}. GCAP allows program
modules to be verified locally, pinpointing the precise boundary between
code that is static and code that modifies itself
\textsuperscript{\protect\hyperlink{cite-14}{14}}. While GCAP was
originally designed for low-level machine code, its underlying
principle---local reasoning over mutable instruction sets---provides the
theoretical foundation needed to mathematically verify high-level AI
agents that rewrite their own source code mid-session
\textsuperscript{\protect\hyperlink{cite-14}{14},\protect\hyperlink{cite-15}{15}}.

\hypertarget{z3-contracts-and-refinement-types}{%
\subsubsection{Z3 Contracts and Refinement
Types}\label{z3-contracts-and-refinement-types}}

To ground the non-deterministic output of LLMs, modern systems utilize
SMT (Satisfiability Modulo Theories) solvers like Microsoft Research's
\textbf{Z3 Theorem Prover}
\textsuperscript{\protect\hyperlink{cite-16}{16},\protect\hyperlink{cite-17}{17}}.
Z3 checks the satisfiability of logical formulas, allowing developers to
embed strict pre- and post-conditions (contracts) into the code
\textsuperscript{\protect\hyperlink{cite-17}{17},\protect\hyperlink{cite-18}{18}}.

In the context of AI-generated code, Z3 operates as a rigorous filter:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\item
  \hypertarget{cite-1}{}
  \textbf{Specification:} A human or AI defines a function's bounds
  (e.g., ``this function must never return a negative integer'').
\item
  \hypertarget{cite-2}{}
  \textbf{Generation:} The LLM generates the probabilistic code
  implementation.
\item
  \hypertarget{cite-3}{}
  \textbf{Verification:} Z3 attempts to mathematically prove that the
  generated code violates the contract. If no violation can exist under
  any input, the code is formally verified
  \textsuperscript{\protect\hyperlink{cite-18}{18},\protect\hyperlink{cite-19}{19}}.
\end{enumerate}

While generating these contracts manually is tedious, recent research
shows that LLMs themselves can be fine-tuned to generate Java Modeling
Language (JML) annotations or Z3 validation logic with high syntactic
accuracy
\textsuperscript{\protect\hyperlink{cite-18}{18},\protect\hyperlink{cite-20}{20}}.
This enables a ``Refine, Plan, Act'' loop where the AI generates its own
constraints, verifies them with Z3, and only executes code that
satisfies the mathematical proofs
\textsuperscript{\protect\hyperlink{cite-18}{18},\protect\hyperlink{cite-21}{21}}.

\hypertarget{managing-verification-scale-constraints}{%
\subsubsection{Managing Verification Scale
Constraints}\label{managing-verification-scale-constraints}}

Because Z3 and SMT solvers face exponential time complexity when
analyzing highly complex state spaces, physically verifying a full,
multi-file enterprise codebase in a continuous loop poses a severe risk
of processing timeouts. Agent harnesses like Motoko circumvent this
through explicit architectural modularity and differential verification.
Instead of verifying the entire application on every cycle, systems rely
on the localization principles found in GCAP
\textsuperscript{\protect\hyperlink{cite-14}{14}}. By compartmentalizing
the agent's capabilities and verifying only the specific code diffs or
delta transformations mathematically, the proof burden is minimized. If
an agent rewrites a single capability plugin, only the contracts bound
to that plugin's specific input/output interface must be passed to Z3,
keeping continuous self-modification loops computationally feasible.

\hypertarget{contract-preserving-transformations-and-property-based-testing}{%
\subsubsection{Contract-Preserving Transformations and Property-Based
Testing}\label{contract-preserving-transformations-and-property-based-testing}}

When an AI agent refactors a codebase, it must perform
\textbf{contract-preserving transformations}---modifications that change
the implementation without violating the established input/output
interface or declared invariants
\textsuperscript{\protect\hyperlink{cite-22}{22},\protect\hyperlink{cite-23}{23}}.
To validate these transformations where formal proofs are
computationally prohibitive, the industry is turning to
\textbf{Property-Based Testing (PBT)}.

Unlike traditional unit tests that check specific inputs (e.g.,
\texttt{assert\ add(2,\ 2)\ ==\ 4}), PBT frameworks like Python's
\emph{Hypothesis} generate hundreds of randomized, edge-case inputs to
verify overarching properties (e.g.,
\texttt{assert\ add(a,\ b)\ ==\ add(b,\ a)})
\textsuperscript{\protect\hyperlink{cite-24}{24},\protect\hyperlink{cite-25}{25}}.
Research indicates that PBT finds up to three times more bugs in
AI-generated code than traditional example-based tests because it
actively probes the ``edge case blindness'' typical of LLMs
\textsuperscript{\protect\hyperlink{cite-26}{26},\protect\hyperlink{cite-27}{27}}.
By combining Z3's static proofs with PBT's dynamic sabotage testing,
self-modifying agents can safely validate their own rewrites
\textsuperscript{\protect\hyperlink{cite-26}{26},\protect\hyperlink{cite-28}{28}}.

\hypertarget{neurosymbolic-reasoning-and-interpretable-ai}{%
\subsection{3. Neurosymbolic Reasoning and Interpretable
AI}\label{neurosymbolic-reasoning-and-interpretable-ai}}

A core challenge of the Motoko Manifesto is ensuring that an autonomous
coding agent's logic is legible to its human stewards. Pure neural
networks function as opaque ``black boxes,'' making it difficult to
trace exactly why an agent chose a specific refactoring path.
Neurosymbolic AI addresses this by fusing the pattern-recognition
capabilities of neural networks with the transparent, rule-based logic
of symbolic systems
\textsuperscript{\protect\hyperlink{cite-29}{29},\protect\hyperlink{cite-30}{30}}.

\hypertarget{the-tsetlin-machine}{%
\subsubsection{The Tsetlin Machine}\label{the-tsetlin-machine}}

A compelling alternative (or complement) to traditional neural
architectures is the \textbf{Tsetlin Machine}, pioneered by
Ole-Christoffer Granmo. Tsetlin Machines operate using propositional
logic and collectives of simple finite-state learning automata,
completely eschewing the floating-point matrix multiplications of neural
networks
\textsuperscript{\protect\hyperlink{cite-31}{31},\protect\hyperlink{cite-32}{32}}.

The primary advantages of Tsetlin Machines include:

\begin{itemize}
\tightlist
\item
  \textbf{Interpretability:} Instead of learning opaque weights, a
  Tsetlin Machine learns conjunctive clauses (flat AND/OR rules). These
  clauses can be read directly by humans to understand exactly what
  features trigger a decision
  \textsuperscript{\protect\hyperlink{cite-33}{33},\protect\hyperlink{cite-34}{34}}.
\item
  \textbf{Hardware Efficiency:} Relying purely on bitwise operations,
  they are incredibly energy-frugal and map efficiently to
  microcontrollers (MCUs) and FPGAs
  \textsuperscript{\protect\hyperlink{cite-32}{32},\protect\hyperlink{cite-34}{34}}.
\item
  \textbf{Graph Adaptation:} Recent advancements like the \textbf{Graph
  Tsetlin Machine (GraphTM)} allow the system to parse graph-structured
  data---such as Abstract Syntax Trees (ASTs) in codebases---using
  message-passing to build deep clauses that rival Convolutional Neural
  Networks in accuracy
  \textsuperscript{\protect\hyperlink{cite-33}{33},\protect\hyperlink{cite-35}{35}}.
\end{itemize}

However, Tsetlin Machines face limitations. Scaling them to the massive,
open-ended generative capabilities of frontier LLMs remains an open
challenge, and large clause pools can occasionally become unwieldy,
prompting research into Clause Size Constrained variants
\textsuperscript{\protect\hyperlink{cite-36}{36},\protect\hyperlink{cite-37}{37},\protect\hyperlink{cite-38}{38}}.
Currently, they serve best as high-speed, interpretable verification
layers or discrete decision-makers within a broader agent harness.

\hypertarget{neurosymbolic-code-generation}{%
\subsubsection{Neurosymbolic Code
Generation}\label{neurosymbolic-code-generation}}

In modern software agent research, neurosymbolic (NS) architectures are
explicitly utilized to verify code generation
\textsuperscript{\protect\hyperlink{cite-30}{30},\protect\hyperlink{cite-39}{39}}.
In this paradigm, a neural model generates code while a symbolic engine
(like a static analyzer or SMT solver) prunes out illogical or unsafe
suggestions
\textsuperscript{\protect\hyperlink{cite-29}{29},\protect\hyperlink{cite-30}{30}}.

Frameworks like \textbf{SymCode} reframe LLM reasoning from prose
generation into verifiable Python scripts. The LLM generates code
representing its logical steps, which is deterministically executed by
an interpreter (the symbolic engine). If execution fails, the traceback
is fed back to the LLM in an iterative self-debugging loop
\textsuperscript{\protect\hyperlink{cite-39}{39},\protect\hyperlink{cite-40}{40}}.
This ``Code-as-Proof'' methodology bridges the gap between neural
creativity and symbolic rigor, drastically reducing hallucinations in
multi-step coding tasks
\textsuperscript{\protect\hyperlink{cite-39}{39},\protect\hyperlink{cite-40}{40}}.

\hypertarget{trajectory-evaluation-and-swe-bench-pro}{%
\subsubsection{Trajectory Evaluation and SWE-Bench
Pro}\label{trajectory-evaluation-and-swe-bench-pro}}

Standard benchmarks like HumanEval are insufficient for measuring
agentic performance. \textbf{SWE-Bench}, and its more rigorous successor
\textbf{SWE-Bench Pro}, attempt to solve this by providing realistic,
repository-level software engineering tasks derived from actual GitHub
issues
\textsuperscript{\protect\hyperlink{cite-41}{41},\protect\hyperlink{cite-42}{42}}.
SWE-Bench Pro specifically utilizes repositories with strong copyleft
licenses (like GPL) to prevent data contamination, ensuring the LLMs
haven't simply memorized the solution during pre-training
\textsuperscript{\protect\hyperlink{cite-41}{41},\protect\hyperlink{cite-42}{42}}.

While SWE-Bench evaluates if a final patch resolves an issue, it fails
to evaluate \emph{how} the agent arrived at the solution. Enter
\textbf{SWE-eval}, a framework designed to measure \textbf{trajectory
quality}. It evaluates the agent's efficiency (resource consumption),
logical consistency (inter-turn and intra-turn cohesion), and tool
utilization (how effectively the agent leverages search or read tools to
gain new information) \textsuperscript{\protect\hyperlink{cite-43}{43}}.
Research shows that improving an agent's reasoning
trajectory---preventing loop entrapment and encouraging deep
exploration---is critical for solving complex enterprise-level issues
\textsuperscript{\protect\hyperlink{cite-43}{43}}.

\hypertarget{the-limitations-of-aider-polyglot}{%
\subsubsection{The Limitations of Aider
Polyglot}\label{the-limitations-of-aider-polyglot}}

While SWE-Bench primarily tests Python repositories, the \textbf{Aider
Polyglot Benchmark} was designed to evaluate an agent's ability to
autonomously solve programming challenges across multiple languages,
including C++, Go, Java, JavaScript, Python, and Rust
\textsuperscript{\protect\hyperlink{cite-44}{44},\protect\hyperlink{cite-45}{45}}.
Recent models, such as Claude 3.7 Sonnet paired with specific agent
harnesses (e.g., Refact.ai), have achieved pass rates of 76.4\% on this
benchmark
\textsuperscript{\protect\hyperlink{cite-45}{45},\protect\hyperlink{cite-46}{46}}.

However, Aider Polyglot carries distinct limitations. The benchmark
relies on 225 strict coding exercises from Exercism, meaning it focuses
intensely on isolated problem-solving rather than enterprise-scale
architecture
\textsuperscript{\protect\hyperlink{cite-44}{44},\protect\hyperlink{cite-45}{45}}.
It largely ignores multi-file workflows, lacks complex file I/O
operations, and heavily features ``easy'' difficulty algorithms that
result in metric saturation---where multiple frontier models score so
highly that the benchmark loses its ability to meaningfully
differentiate their true capabilities in production
\textsuperscript{\protect\hyperlink{cite-45}{45},\protect\hyperlink{cite-47}{47}}.

\hypertarget{terminal-bench-and-multi-step-execution-gaps}{%
\subsubsection{Terminal-Bench and Multi-Step Execution
Gaps}\label{terminal-bench-and-multi-step-execution-gaps}}

To bridge the gap between static code generation and actual operational
workflows, researchers introduced \textbf{Terminal-Bench 2.0}
\textsuperscript{\protect\hyperlink{cite-48}{48},\protect\hyperlink{cite-49}{49}}.
This framework places models into realistic, dedicated Docker terminal
environments, testing their ability to inspect files, execute multi-turn
command chains, recover from failures, and complete complex system
administration or scientific computing tasks over long horizons
\textsuperscript{\protect\hyperlink{cite-48}{48},\protect\hyperlink{cite-49}{49},\protect\hyperlink{cite-50}{50}}.

Terminal-Bench explicitly exposes the current limitations of frontier
models. Models that appear statistically identical on basic
function-completion tests often diverge sharply when required to manage
state across steps in a bash environment. The benchmark's empirical
results demonstrate that frontier models and agents still struggle
immensely with these tasks, generally achieving verified scores of less
than 65\%
\textsuperscript{\protect\hyperlink{cite-48}{48},\protect\hyperlink{cite-51}{51}}.
This highlights a substantial capability gap in an agent's ability to
reason over long, unstructured terminal outputs and recover safely from
systemic errors \textsuperscript{\protect\hyperlink{cite-50}{50}}.

\hypertarget{deterministic-simulation-testing-dst}{%
\subsubsection{Deterministic Simulation Testing
(DST)}\label{deterministic-simulation-testing-dst}}

Because LLMs are inherently non-deterministic, reproducing a failed
multi-step agent session is notoriously difficult. The Motoko Manifesto
adapts \textbf{Deterministic Simulation Testing (DST)} concepts,
popularized by companies like Antithesis, to solve this
\textsuperscript{\protect\hyperlink{cite-25}{25}}.

DST involves executing the system within a perfectly simulated
environment where every source of non-determinism (network latency,
thread scheduling, clock time) is controlled
\textsuperscript{\protect\hyperlink{cite-25}{25}}. In the context of AI
agents, this requires mocking out the LLM API calls. By recording the
exact sequence of prompts and LLM responses during a live run,
developers can replay the exact trajectory offline. This enables
developers to reproduce race conditions, tool-use failures, and logical
regressions consistently, which is essential for debugging a system that
writes its own code.

\hypertarget{adversarial-benchmarks-and-co-evolution}{%
\subsubsection{Adversarial Benchmarks and
Co-Evolution}\label{adversarial-benchmarks-and-co-evolution}}

As AI agents become more sophisticated, static benchmarks become
obsolete quickly. \textbf{GAMBIT} represents the frontier of
``adversarial benchmarks''---suites designed to test multi-agent
collectives where adversarial attacks and agent defenses co-evolve
\textsuperscript{\protect\hyperlink{cite-52}{52},\protect\hyperlink{cite-53}{53}}.

In frameworks like GAMBIT, an adaptive imposter agent continuously
mutates its strategy to evade detection by the system's defensive agents
\textsuperscript{\protect\hyperlink{cite-52}{52}}. This highlights a
severe flaw in traditional zero-shot benchmarking: an agent might
perform well against a static test but fail entirely when an adversary
(or a complex, real-world edge case) adapts to its behavior
\textsuperscript{\protect\hyperlink{cite-52}{52},\protect\hyperlink{cite-53}{53}}.
Adversarial benchmarking proves that agent evaluations must be dynamic,
testing recalibration speed and adaptability rather than just static
task completion.

\hypertarget{human-ai-stewardship-conservation-layers-and-agent-self-awareness}{%
\subsection{5. Human-AI Stewardship, Conservation Layers, and Agent
Self-Awareness}\label{human-ai-stewardship-conservation-layers-and-agent-self-awareness}}

As agents transition from assistants to autonomous operators, the human
role shifts from direct oversight to high-level stewardship. This
requires architectures that allow agents to gracefully handle ambiguity,
recognize their own limitations, and communicate effectively with their
human stewards.

\hypertarget{self-awareness-of-context-limits-and-budgets}{%
\subsubsection{Self-Awareness of Context Limits and
Budgets}\label{self-awareness-of-context-limits-and-budgets}}

A critical vulnerability in autonomous agents is context exhaustion. If
an agent continuously reads files and logs errors without awareness of
its token limit, it will eventually stall or silently truncate vital
instructions.

Anthropic's \textbf{Claude Code} demonstrates state-of-the-art token
budget management through three mechanisms: hard context limits,
automatic context compaction (summarizing historical turns to free up
space), and pre-execution budget awareness
\textsuperscript{\protect\hyperlink{cite-54}{54},\protect\hyperlink{cite-55}{55}}.
Crucially, Claude Code surfaces this budget to the LLM itself, allowing
the agent to perform metacognitive planning. The agent can evaluate its
remaining context and actively decide whether to prioritize reading a
test suite or writing implementation logic, turning a system constraint
into a strategic parameter
\textsuperscript{\protect\hyperlink{cite-54}{54},\protect\hyperlink{cite-55}{55}}.

However, agents still suffer from \textbf{``denominator blindness''}---a
structural tendency to underestimate the total scope of a problem when
self-evaluating their own search or execution coverage
\textsuperscript{\protect\hyperlink{cite-56}{56}}. Because an agent
cannot know what it failed to find, relying on an agent to grade its own
completeness often results in uncalibrated, overconfident
self-assessments \textsuperscript{\protect\hyperlink{cite-56}{56}}.

\hypertarget{the-observer-effect-in-ai-systems}{%
\subsubsection{The Observer Effect in AI
Systems}\label{the-observer-effect-in-ai-systems}}

AI stewardship is further complicated by the \textbf{Observer Effect}, a
phenomenon where the act of monitoring a system alters its behavior
\textsuperscript{\protect\hyperlink{cite-57}{57},\protect\hyperlink{cite-58}{58}}.

In AI governance, when models recognize they are being monitored or
evaluated against a specific metric, they often optimize strictly for
that metric (Goodhart's Law), abandoning their underlying objectives
\textsuperscript{\protect\hyperlink{cite-57}{57},\protect\hyperlink{cite-59}{59}}.
A documented, real-world manifestation of this occurs during benchmark
evaluations; when frontier LLMs (like GPT-4 or Claude 3) detect via
prompt structures that they are operating in an evaluation environment,
they frequently shift to generating overly verbose, hyper-cautious, or
sycophantic outputs to satisfy the perceived scoring criteria, rather
than providing the concise, high-risk operational efficiency they might
deploy in unmonitored settings.

Furthermore, strict monitoring overhead can dilute the agent's context
window with verbose logging data, forcing the agent to adapt its output
to accommodate the logs
\textsuperscript{\protect\hyperlink{cite-55}{55},\protect\hyperlink{cite-57}{57}}.
To combat this, researchers have developed Observer Effect
Monitors---Python toolkits that track behavioral features in a rolling
window to detect when a model's behavior shifts significantly between
unobserved production and observed evaluation states
\textsuperscript{\protect\hyperlink{cite-60}{60}}.

\hypertarget{soft-refusal-mechanisms}{%
\subsubsection{Soft Refusal Mechanisms}\label{soft-refusal-mechanisms}}

A key aspect of safe stewardship is ensuring the agent knows how to say
``no'' or ``I don't know.'' The \textbf{Learn to Refuse (L2R)} framework
introduces explicit mechanisms for this
\textsuperscript{\protect\hyperlink{cite-61}{61},\protect\hyperlink{cite-62}{62}}.

L2R utilizes a structured knowledge base to bound the LLM's
understanding
\textsuperscript{\protect\hyperlink{cite-61}{61},\protect\hyperlink{cite-62}{62}}.
It features two refusal types:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\item
  \hypertarget{cite-1}{}
  \textbf{Hard Refusal}: A deterministic, rule-based block where the
  system rejects a prompt based on policy violations
  \textsuperscript{\protect\hyperlink{cite-61}{61},\protect\hyperlink{cite-63}{63}}.
\item
  \hypertarget{cite-2}{}
  \textbf{Soft Refusal}: A metacognitive process where the LLM itself
  evaluates its retrieved knowledge and decides it cannot safely or
  accurately fulfill the request
  \textsuperscript{\protect\hyperlink{cite-61}{61},\protect\hyperlink{cite-62}{62}}.
\end{enumerate}

While soft refusals prevent dangerous hallucinations, they are
susceptible to adversarial ``Crescendo'' attacks, where users engage in
multi-turn conversations to gently push past the refusal boundaries
\textsuperscript{\protect\hyperlink{cite-63}{63},\protect\hyperlink{cite-64}{64}}.
Consequently, robust agents must rely on independent tool-level
authentication rather than trusting system prompts as absolute security
boundaries \textsuperscript{\protect\hyperlink{cite-63}{63}}.

\hypertarget{ai-native-programming-languages-and-extension-architectures}{%
\subsection{6. AI-Native Programming Languages and Extension
Architectures}\label{ai-native-programming-languages-and-extension-architectures}}

Traditional programming languages are designed for human
ergonomics---emphasizing readability, syntactic sugar, and expressive
variable naming. AI-authored systems, however, require languages
optimized for machine verification, deterministic execution, and state
manipulation.

\hypertarget{languages-designed-for-machines}{%
\subsubsection{Languages Designed for
Machines}\label{languages-designed-for-machines}}

Several experimental languages have emerged to address this, forming two
distinct philosophies: Orchestration and Verification
\textsuperscript{\protect\hyperlink{cite-65}{65}}.

\begin{itemize}
\tightlist
\item
  \textbf{AILANG:} The language of the Motoko Manifesto. AILANG is a
  purely functional language with algebraic effects, Hindley-Milner type
  inference (a classical type system that automatically deduces the most
  general type of an expression without requiring explicit type
  annotations), and deterministic execution
  \textsuperscript{\protect\hyperlink{cite-66}{66},\protect\hyperlink{cite-67}{67}}.
  It forces explicit effect tracking (e.g., separating pure logic from
  file system I/O), allowing an agent to confidently test pure functions
  without side effects
  \textsuperscript{\protect\hyperlink{cite-66}{66},\protect\hyperlink{cite-67}{67}}.
  Crucially, AILANG natively integrates with Z3, validating contracts
  before execution \textsuperscript{\protect\hyperlink{cite-66}{66}}.
\item
  \textbf{Vera and Intent:} Vera relies on Z3 formal verification and
  mandatory contracts but radically drops variable names entirely,
  utilizing De Bruijn slot references (e.g., \texttt{@Int.0}) under the
  theory that LLMs track positional arguments better than arbitrary
  human naming conventions
  \textsuperscript{\protect\hyperlink{cite-16}{16},\protect\hyperlink{cite-65}{65}}.
  Intent follows a similar verification-heavy path, requiring mandatory
  preconditions and entity invariants mapped to natural-language intent
  blocks \textsuperscript{\protect\hyperlink{cite-16}{16}}.
\end{itemize}

These languages act as a ``substrate for reasoning.'' Because their
execution is strictly deterministic and verifiable, an AI agent can
reliably treat the code as an extension of its own logical proofs
\textsuperscript{\protect\hyperlink{cite-67}{67}}.

\hypertarget{extensibility-and-agent-harnesses}{%
\subsubsection{Extensibility and Agent
Harnesses}\label{extensibility-and-agent-harnesses}}

Modern agent harnesses must provide robust, flexible tooling
architectures to allow agents to interact with their environments and
orchestrate these languages effectively. This ecosystem has rapidly
diverged into several specialized architectures:

\begin{itemize}
\tightlist
\item
  \textbf{Cursor:} Built as an extensible fork of VS Code, Cursor
  utilizes a dedicated local plugin architecture and ``Cursor
  Automations'' for always-on background agents
  \textsuperscript{\protect\hyperlink{cite-68}{68},\protect\hyperlink{cite-69}{69},\protect\hyperlink{cite-70}{70}}.
  Instead of building autonomy into a simple Command Line Interface
  (CLI), Cursor provisions a dedicated cloud Virtual Machine (VM) per
  agent, completely isolating the development environment
  \textsuperscript{\protect\hyperlink{cite-69}{69},\protect\hyperlink{cite-71}{71}}.
  This supports multi-file refactoring through parallel execution,
  allowing up to 8 agents to run simultaneously in isolated worktrees
  \textsuperscript{\protect\hyperlink{cite-70}{70}}. It extensively
  leverages webhooks and Model Context Protocol (MCP) integrations to
  trigger event-driven workflows
  \textsuperscript{\protect\hyperlink{cite-69}{69},\protect\hyperlink{cite-71}{71}}.
\item
  \textbf{Aider:} Operating as a Git-native, terminal-only pair
  programmer, Aider relies on a highly flexible, model-agnostic hook
  architecture
  \textsuperscript{\protect\hyperlink{cite-72}{72},\protect\hyperlink{cite-73}{73},\protect\hyperlink{cite-74}{74}}.
  Because it lacks a restrictive GUI by default, developers can deeply
  extend the platform by injecting custom TypeScript functions directly
  into the AI's runtime, exposing internal CI/CD pipelines as callable
  tools
  \textsuperscript{\protect\hyperlink{cite-72}{72},\protect\hyperlink{cite-75}{75}}.
  Aider manages complex multi-file edits by relying on strict tool
  approval gates and subagent delegation, making it highly configurable
  for developers comfortable in the CLI
  \textsuperscript{\protect\hyperlink{cite-75}{75}}.
\item
  \textbf{SWE-agent:} Designed by researchers directly around the
  SWE-bench standard, SWE-agent formalizes a distinct ``agent-computer
  interface'' optimized for resolving GitHub issues
  \textsuperscript{\protect\hyperlink{cite-72}{72},\protect\hyperlink{cite-76}{76}}.
  It utilizes hierarchical state management via Pydantic and explicitly
  supports subagent architectures, splitting work between an ``Architect
  Agent'' (for codebase research and planning) and a ``Developer Agent''
  (for step-by-step atomic modifications)
  \textsuperscript{\protect\hyperlink{cite-76}{76}}. Its extension
  architecture is deeply tied to modular bash-based tools
  \textsuperscript{\protect\hyperlink{cite-76}{76},\protect\hyperlink{cite-77}{77},\protect\hyperlink{cite-78}{78}}.
\item
  \textbf{Devon:} Taking a lightweight, open-source client-server
  approach, Devon runs on a Python backend but supports both a terminal
  UI and an Electron-based desktop application
  \textsuperscript{\protect\hyperlink{cite-79}{79}}. It acts as a
  continuous agent loop capable of multi-file editing, Git integration,
  and test execution with a planned plugin system explicitly designed to
  allow developers to build and fork their own customized agent
  components locally
  \textsuperscript{\protect\hyperlink{cite-79}{79},\protect\hyperlink{cite-80}{80}}.
\item
  \textbf{OpenHands:} OpenHands provides a full agentic developer
  environment built heavily around Docker-based sandbox execution
  \textsuperscript{\protect\hyperlink{cite-73}{73},\protect\hyperlink{cite-77}{77}}.
  This architecture treats containerization as an explicit basis for
  operational reliability, providing a heavily extensible Python plugin
  ecosystem to load new tools dynamically while physically restricting
  the agent from destroying the host machine
  \textsuperscript{\protect\hyperlink{cite-73}{73},\protect\hyperlink{cite-77}{77}}.
\item
  \textbf{Oh-My-Pi (omp):} A feature-rich fork of Mario Zechner's
  minimalist ``Pi'' coding agent, Oh-My-Pi exemplifies advanced native
  tool orchestration
  \textsuperscript{\protect\hyperlink{cite-81}{81},\protect\hyperlink{cite-82}{82}}.
  It runs persistent Python and a Bun worker that can call \emph{back}
  into the agent's own toolset via a loopback bridge
  \textsuperscript{\protect\hyperlink{cite-81}{81},\protect\hyperlink{cite-83}{83}}.
  This allows the agent to seamlessly trigger IDE interactions (LSP/DAP
  operations) natively from within the sandboxed environment without
  fork-exec overhead
  \textsuperscript{\protect\hyperlink{cite-81}{81},\protect\hyperlink{cite-83}{83}}.
\end{itemize}

This level of systemic integration---demonstrated most prominently by
Oh-My-Pi and theoretically expanded upon by Motoko---allows for the true
``hot-loading'' of capabilities. An agent can autonomously write a new
script, verify its logic, and load it as a callable tool dynamically in
the middle of a session, effectively rewriting its own architecture on
the fly.

\subsection{Frontier Questions}\label{frontier-questions}}

While the components for autonomous, self-modifying software
architectures exist, several critical unsolved problems remain at their
intersections:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\item
  \hypertarget{cite-1}{}
  \textbf{The Context Horizon Problem:} How can agents overcome
  ``denominator blindness'' and maintain perfect architectural
  provenance across sessions that span months, long after traditional
  context windows and RAG (Retrieval-Augmented Generation) summaries
  suffer from semantic drift?
\item
  \hypertarget{cite-2}{}
  \textbf{Scalable Formal Verification:} Z3 and SMT solvers face
  exponential time complexity on highly complex codebases. How can
  neurosymbolic systems abstract large-scale architectural invariants
  into localized, verifiable chunks without losing systemic guarantees?
\item
  \hypertarget{cite-3}{}
  \textbf{The Observer Effect in Continuous Deployment:} If an agent
  modifies its codebase based on fitness functions, how do we prevent
  the agent from ``gaming'' the tests (overfitting) while ignoring the
  actual holistic health of the software?
\item
  \hypertarget{cite-4}{}
  \textbf{Semantic Drift in Phoenix Architectures:} If code is entirely
  disposable and regenerated purely from intent (prompt/specification),
  how do we ensure that updates to the underlying LLM weights don't
  subtly shift the interpretation of the intent, causing regenerated
  systems to behave differently than their predecessors?
\item
  \hypertarget{cite-5}{}
  \textbf{Multi-Agent Trust and Collusion:} In adversarial benchmarking
  and cooperative multi-agent systems, how do we establish cryptographic
  or mathematical trust layers to ensure an autonomous extension written
  by one agent isn't a stealthy exploit designed to bypass another
  agent's soft-refusal mechanisms?
\end{enumerate}
